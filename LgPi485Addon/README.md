# LG PI485 에어컨 애드온

EW11 RS485-to-TCP 브리지를 통해 LG PI485 에어컨을 Home Assistant에 연동하는
MQTT 애드온입니다.

프로토콜 판단은 공개된 LG PI485 설치/중앙제어 문서와 현장 패킷 캡처를 기준으로
정리했습니다.

## 현재 범위

- EW11 TCP 연결 관리
- MQTT 연결 및 EW11 연결 진단
- Home Assistant MQTT discovery
- 실내기 상태 주기 폴링
- 실내기별 MQTT climate 엔티티
- 실내기별 전원 및 제어 잠금 스위치
- 배관 온도, raw frame, 에러 코드, LGAP 부하 진단 센서
- 추후 프로토콜 분석을 위한 알 수 없는 패킷 캡처
- 냉방 전용 기본값: `off,cool,fan_only,dry,auto`

초기 튜닝 단계에서는 대부분의 진단 엔티티를 기본 활성 상태로 둡니다. 단,
`LG PI485 EW11 마지막 수신 시간`은 값이 자주 바뀌고 EW11 통신 상태 확인용에
가까워서 기본 비활성으로 둡니다.

운영 로그는 제어 요청과 제어 응답 중심으로 남깁니다. 주기적인 상태 요청/응답
로그는 기본 비활성이며, 패킷 흐름을 자세히 확인해야 할 때만
`log_status_polling`을 켭니다.

상태 폴링 기본값은 2초 주기이며, 실내기별 요청은 150ms 간격으로 큐에 넣습니다.
제어 명령이 들어오면 대기 중인 상태 요청보다 제어 명령을 우선 처리합니다.

EW11 연결이 장시간 끊어진 경우 기본적으로 30초 간격으로 20회 재시도한 뒤
애드온을 종료합니다. 계절 종료처럼 에어컨 제어 장비를 오래 내려두는 상황에서
재연결 대기 프로세스가 계속 남지 않게 하기 위한 동작입니다. 계속 실행 상태로
두고 싶다면 `ew11_exit_on_max_retries`를 `false`로 설정합니다.

상태 파일에는 발견된 실내기 목록만 저장합니다. 실시간 에어컨 상태는 MQTT retained
메시지와 메모리 상태로 유지하며, 상태 폴링을 받을 때마다 `/share`에 파일을 다시
쓰지 않습니다.

## Home Assistant 엔티티

발견된 각 실내기에는 climate 엔티티와 보조 엔티티가 생성됩니다.

- climate: 모드, 목표 온도, 현재 온도, 팬 모드, 스윙 모드
- switch: 전원, 제어 잠금
- binary sensor: 전원, 잠금, 공기청정, 존 가동
- diagnostic sensor: 배관 온도, 에러 코드, 부하 필드
- text sensor: 16바이트 LGAP raw frame

초기에는 실제 패킷과 Home Assistant 상태를 계속 비교할 수 있도록 진단값을
노출해둡니다. 아직 의미가 완전히 검증되지 않은 값은 일반 사용자용 센서로
올리지 않고 raw 또는 진단 데이터로 유지합니다.

## 확인된 한계

아래 내용은 현장 패킷 캡처와 공개된 LG 중앙제어 문서를 함께 보고 정리한
내용입니다.

- 목표 온도는 1도 단위로 노출합니다.
- Wi-Fi 또는 ThinQ 앱에서는 `0.5°C` 단위가 보일 수 있지만, 관측된 PI485/LGAP
  상태 프레임은 `27.0`과 `27.5`, `28.0`과 `28.5`를 구분하지 않았습니다.
- 습도는 관측된 16바이트 PI485/LGAP 상태 프레임에 노출되지 않습니다.
- 현재 실내 온도는 LGAP 프레임 기준이라 Wi-Fi 앱 표시와 약간 다를 수 있습니다.
- 기본 설정은 냉방 전용이므로 난방 모드는 비활성입니다.
- 실제 압축기 동작 상태는 직접 노출되지 않습니다. 동작 상태 추론값이 필요하다면
  보수적으로 다뤄야 합니다.

## 냉방 및 난방

난방 모드는 기본적으로 비활성입니다.

```yaml
cooling_only: true
supported_hvac_modes: "off,cool,fan_only,dry,auto"
```

난방을 지원하는 설치 환경에서는 다음처럼 설정합니다.

```yaml
cooling_only: false
supported_hvac_modes: "off,cool,fan_only,dry,heat,auto"
```

## 프로토콜 메모

요청 프레임은 8바이트입니다.

```text
tx_byte_0 00 request_id zone control_flags mode_fan_swing target_temp checksum
```

기본값은 `tx_byte_0 = 0x80`, `request_id = 0xA3`입니다. 일반 애드온 설정에서는
노출하지 않고 코드 기본값으로 사용합니다.

응답 프레임은 16바이트이며 `0x10`으로 시작해야 합니다. 체크섬은 체크섬 바이트를
제외한 모든 바이트로 다음과 같이 계산합니다.

```text
(sum(bytes) & 0xFF) ^ 0x55
```

주요 응답 필드는 다음과 같습니다.

- byte 4: 실내기 주소
- byte 5: 에러 코드
- byte 6: 모드, 팬, 스윙
- byte 7: 목표 온도 raw 값
- byte 8: 실내 온도 raw 값
- byte 9: 배관 1 온도 raw 값
- byte 10: 배관 2 온도 raw 값
- byte 11: 존 현재 부하
- byte 12: 존 전원 상태값
- byte 13: 존 설계 부하
- byte 14: 실외기 총 부하

현재 캡처 기준으로는 byte 11을 온도값이라고 단정하기 어려워 byte 11-14 값을
부하 계열 진단 센서로 노출합니다.

## 명령 처리 방식

제어 명령이 처리 중일 때는 상태 폴링을 잠시 멈춥니다. 제어 명령은 일반 상태
요청보다 앞에 배치되며, 기대한 응답을 받거나 재시도 한도에 도달하기 전까지
추가 폴링 프레임을 보내지 않습니다. EW11만 단독으로 물려서 애드온이 PI485 버스의
능동 제어자 역할을 하는 현재 구성에서 특히 중요합니다.

잠금 처리는 보수적으로 동작합니다. 일반 전원, 모드, 온도 명령은 기존 잠금 비트를
자동으로 보존해서 싣지 않습니다. 잠금 변경은 잠금 스위치 명령으로만 전송합니다.

`log_status_polling`은 기본값이 `false`입니다. 이 상태에서는 일반 상태 폴링
요청/응답은 로그에 남기지 않고, 실제 제어 요청과 그 응답만 남깁니다.
